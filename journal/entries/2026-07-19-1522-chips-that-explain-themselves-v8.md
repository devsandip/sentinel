# Chips that explain themselves, and OPA is off the table

2026-07-19 15:22. Previous: [2026-07-19-1354-chrome-that-tells-the-truth-v7.md](2026-07-19-1354-chrome-that-tells-the-truth-v7.md)

Two things happened. A gap closed that had been open since the control catalogue was written, and Sandip finally called OPA.

The gap first. A control chip sometimes explained itself on click and sometimes did nothing, and there was no rule for which. `_control_popover` has been the right mechanism for a while: it reads the 27-entry ControlInfo catalogue, shows the name, the action type, the plain-language what, the why it exists, and an "In this run" line assembled from the published stage output. It was already wired at three call sites. But thirty lines above it, `_engine_bar` was building `<span class='ctlchip'>CTL-EGRESS-01</span>` as static markdown. Every stage of the walkthrough rendered eight of those, next to working code that could have made them real. The topbar chips, the import allowlist, the certification cards and all three catalog tables had no disclosure at all.

The rule I settled on, now written into ui-spec 4.3: a chip is clickable when it names a governance decision, inert when it names a fact. Control ids, a dataset's classification, a model's promotion status all open the catalogue entry for the control that made the call. Module names in the import allowlist, row counts, licenses do not. There is no catalogue entry behind `numpy`, and minting a `CTL-` id so a chip has something to open is exactly the theatre this project exists to argue against.

That rule did most of the design work. The import allowlist was the case I expected to be hard and was not. Clicking `os` should not explain `os`; it should explain why `os` is denied. So the deny list is now grouped by the control that denies it, mirroring `import_verdict`'s own precedence (egress, then filesystem, then dynamic code), and each row carries one chip for that control. Four popovers instead of thirty-six, and the grouping is the honest structure rather than a flat red blob.

The topbar chips both open `CTL-PURP-01`. That looked redundant at first and is not: the purpose matrix is dataset-by-purpose and those two chips name its two axes, so one control genuinely explains both. The Data chip adds one factual line, read off the real matrix and the classification ceiling, not written by hand. Click it on the dataset registry and berka says it is Confidential, that alone caps the run at L1, and lists the four purposes permitted on it.

One place is deliberately not clickable: the chips inside the global Controls popover. Streamlit's own guidance for `st.popover` is not to nest popovers, and that surface is the see-everything-at-once catch-all anyway. A menu inside a menu reads badly. It is the complement to per-instance disclosure, not a competitor.

Then the tables. Datasets and both Registry tables were `st.dataframe`, which renders every cell as plain text. It cannot carry the `.cls` and `.badge` chips the spec documents, let alone a popover. Sandip asked for the full rebuild rather than a middle ground, so all three are now a header band plus one `st.columns` row per record. Classification, fairness, status and promotion finally read as chips. The cost is real and I wrote it down rather than glossing it: dataframe sorting and column resizing are gone. At eight, three and four rows that is a fair trade, and it would not be at fifty.

Scoping that table work turned up something worse than the chip gap. The dataset registry had no classification column at all. Eight datasets, each with a classification that sets the autonomy ceiling, and the table that inventories them did not show it, despite ui-spec 3.4 listing it. It is there now, with the class-count breakdown row above it that the spec also asks for. Two implemented controls, `CTL-CODE-00` and `CTL-DISC-03`, were also missing from their stages' engine lists, so they had no clickable home anywhere.

A knock-on I did not plan. Making the topbar chips into popovers forced the bar out of a single markdown blob, because a popover cannot live inside raw HTML. The old 9:3 `st.columns` split went with it, in favour of the mockup's actual structure: one flex row, brand left, everything else right, each item sized to its content. That fixed a clipping bug I had not registered as a bug. The identity trigger had been truncating to "Acting as: Data Sci..." for as long as it has been in the header.

Deployed. Bundle `sentinel-20260719-151623.zip`, CloudFormation applied, EB Green, live-LLM on. Verified on the live site the way that counts: the Data chip opens the real CTL-PURP-01 entry, the dataset registry shows eight rows with eight clickable classification chips and zero surviving dataframes. 382 tests, ruff clean.

One deploy note for next time. The deploy script reads the live-LLM key from `$REPO_ROOT/.env`, and `.env` is gitignored so it exists only in the main checkout, not in any worktree. Deploying from a worktree without passing the key through would have set the CFN parameter to empty and silently flipped prod back to scripted narration. Health would still be green and the site would still load. I passed it through from the main checkout. Anyone deploying from a worktree needs to do the same.

Now OPA. Sandip's words: not in scope for the foreseeable future. That closes the one genuine fork that has been open since v4 and was, as of this morning's entry, still waiting on him. It is not a rejection of the idea. Externalising policy to OPA needs an external server, and the demo already shows the policy logic working in-process: the purpose matrix refuses at Access, the tier resolves as the lower of two ceilings, the gate reads code with real parsers. What OPA would buy is where the policy lives, not whether it exists. That is an architecture decision for a real deployment, not for a credibility artifact.

Moving it from open questions to ruled out. The related open question about the small-cell floor goes with it: making `n < 10` a per-domain policy value was the strongest argument for OPA, and with OPA out, the floor stays a parameter on the Screen with a sane default. That is the right shape regardless.

What is left is smaller than it was this morning. Dark mode, RBAC-gated navigation and the B-style contextual drawers are still deferred by choice. Drift monitoring still has no stage in the lifecycle, which is now the largest genuine hole. The SR 11-7 retrieval still ranks the internal modeling standard above SR 11-7 itself. And `app.py` is past two thousand lines with a hand-rolled router; today added a hundred more in table layout. Streamlit's native multipage support is still the noted fix and it is getting harder to keep deferring.
